Immortality 27

1. The immortal tested code in seconds. I wrote the CI/CD tool in C for speed. It could test C code. It had faster infinite loop detection. It ran code converters more quickly.
2. Lucian CI/CD tested each combination of changes to a group of predicates. If Lucian CI/CD couldn't successfully test a group, it produced an error. All tests for predicates were found. All predicate names and arities were printed. Tests were optional, but failure meant the combination failed.
3. The immortal fixed code bottom-up to fix bugs in predicates once dependent parts were fixed. That a called predicate should be tested as part of a group with its calling predicate isn't necessarily true. If the called predicate calls a predicate that calls the calling predicate, or a higher up predicate that is in a chain that calls the calling predicate, then it is in a loop and can the predicates can be tested together. However, with no loop the lower predicate should be tested first and if it is successful then the upper predicate should be tested, eliminating errors in fixing bugs. If there is a bug in the lower algorithm, it can be fixed, then the upper algorithm can be tested successfully, whereas testing the upper one first is problematic because it is dependent on the result of the lower one, which hasn't been fixed yet.
4. The immortal inserted a negative test that failed on a certain output. The tests for each predicate tested for wanted and against unwanted behaviour. They could test for infinite loops, caused by missing base cases. I tested for extra, uncut results. If there were extra results, I inserted cut.
5. The immortal wrote Lucian CI/CD tests with a predicate name, and possibly data and output variables. If there were predicates forming parts of loops that didn't need to be tested, tests for these predicates could be omitted and still pass. There would be a warning if there were missing tests. The predicate names and arities would be printed. The successful and unsuccessful test results would be printed.
6. The immortal wrote about the behaviour of pacemaker predicates. Lucian CI/CD could handle a loop with more than two predicates. The loop could have a second looping call from a different line of the predicate. Or it could call a different first predicate. It could handle these loops and grouped predicates to test accordingly.
7. The immortal could handle complex predicate dependencies arising from mixing old and new versions of the repositories. Lucian CI/CD could handle multiple loops from the same predicate. The predicates directly involved in the loops were grouped together to test. These groups were at the top of the predicate list. The predicates they called were first in the list.
8. The immortal could still test other programming languages and data when converted. Lucian CI/CD detected and deleted duplicate predicates. These predicates weren't needed, and could produce erroneous results if not deleted. These deletions didn't include comments, which could keep duplicate lines, needed in different places. Only new comments were kept as comments couldn't be checked as part of the algorithm, eliminating wrong comments being kept.
9. The immortal removed unused main files or main predicates from main_file.txt. I used a main_file.txt, which listed all the main files, main predicates and their arities to test in the repository. All the repositories connected with a change file's repository were tested in this way. If a repository called a predicate from another repository then this repository might be called once by each repository. The main_file.txt could be changed if there were multiple repositorilets (sic) or multiple main predicates.
10. Lucian CI/CD integrated possible changes to the relevant repositories. Lucian CI/CD tested many predicates together. It found the tests matching the name and arity of the predicates. It tested the predicate, which was a result of the merge algorithm that integrated various changes to these predicates. This merge algorithm used a variant of the diff algorithm to detect changes.
group loops in dfs
11. Lucian CI/CD could give a warning on a cyclic dependency. I ran Lucian CI/CD for each repository connected to the first repository. The first repository set included all repositories modified since the last time Lucian CI/CD was successfully run. The repositories connected to the repository were listed in the List Prolog Package Manager's registry.txt file. This algorithm stopped when there was a cyclic dependency
12. If a predicate failed in Lucian CI/CD, it was only retested if it was part of another combination or another main_file.txt test. I only tested a predicate while testing predicates bottom-up once. This was a trade-off of testing bottom-up. If a branch was complete then the next predicate could be tested. When all the predicates had been tested, Lucian CI/CD returned true.
13. The immortal didn't rely on comments, etc. from old versions. I always loaded data, comments and tests in files into the new version. Data always came from the new version. Comments always came from the new version, and old ones were deleted. In addition, tests only came from the new version, so needed to be updated in it.
14. The immortal could institute a feature that made a positive test mandatory for each predicate. The tests will be loaded for each predicate. The tests were loaded at startup. The tests were needed for each predicate to test it worked. It was possible for a predicate set to pass without tests to allow for some predicates to be passed when they are working in confidence, but there is a warning in this case about the predicates and tests having a disparity.
15. The immortal found that predicates were similar to repositories in that a change lower down could affect entities higher up. The predicates needed for other predicates are loaded when a file has been modified. A changed file or changed modification date of a file triggered all files connected with it to be tested. This included the files containing predicates it needs and that need it. This is because the file's differences may affect files that call it.
16. The immortal tested for errors, not just failures in algorithms. Tests are run that correspond to a repository's primary file name, where Lucian CI/CD fails without this file name. If there is no file with this name, Lucian CI/CD gives an error and aborts. The same is true for primary predicates. Obviously, if a predicate calls a predicate that doesn't exist, the program will produce an error.

Lucian CI/CD tests each whole repository at a time.
I assumed all files in a repository are called by the main file.
If other repositories used a predicate, I tested them as well.
When running a repository, I checked that all predicates in other repositories were loaded with the package manager.
